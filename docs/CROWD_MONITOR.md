# 混雑状況モニター構想（カメラ×AI×混雑ランプ）

> 🧭 **新しい加盟店にこの機能をセットアップする手順は [MERCHANT_ONBOARDING.md] の STEP 7** にまとめてある（Authenticationでユーザー作成 → `shopOwners/{uid}` に `shopId` を登録 → お店にURL・メール・パスワードを渡す）。

2026-07-14 masaさん発案。お店の防犯カメラ映像をAIが監視し、**混雑状況を赤／黄／緑のランプ**でアプリに表示する仕組み。

**結論：作れる。しかも既存の構成にきれいに乗る。ただし"技術"より"プライバシー設計"と"誤判定リスク"が本丸。**

---

## 1. 全体像

```
[店内カメラ] → [店内のPC/小型端末で"ローカル"AI判定] → 人数だけ送信 → [Firestore]
                     ↑ 映像はここから外に出さない                        ↓
                                                        [ながおか探検録アプリ]
                                                         お店一覧に 🔴🟡🟢 を表示
```

### ⭐ 設計の核心：**映像を絶対に外に出さない**
- AI判定は**店内の端末の中だけ**で完結させる
- サーバーに送るのは **`{ 店ID, 混雑レベル, 更新時刻 }` だけ**（＝ただの数値。映像も顔も送らない）
- **これが法務・信頼・コストの3つを同時に解決する**（詳細は §4）

---

## 2. 技術構成（現実的な最小構成）

### 店舗側
- **入力**：既存の防犯カメラ（多くのIPカメラは **RTSP** で映像を取れる）／無ければUSBウェブカメラでも可
- **端末**：お店の既存PC（常時起動できるなら）／無ければ **Raspberry Pi 5（2〜3万円）** を置く
- **AI**：**人物検出**（YOLOv8n 等の軽量モデル）をローカル推論
  - **顔認識はしない**。「人が何人写っているか」だけを数える
  - **1〜5分に1回**判定すれば十分（混雑度に秒単位の精度は不要）→ CPUでも余裕、電気代もわずか
- **判定**：`人数 ÷ その店の満席人数` で割合を出す
  - 〜40%：🟢 空いている
  - 40〜75%：🟡 混み始め
  - 75%〜：🔴 混雑
  - ※ 閾値は**店ごとに設定**（10席の店と50席の店を同じ数字で判定できない）
- **送信**：数分おきに Firestore に `{ level, count, updatedAt }` を書き込むだけ

### サーバー側
- **Firebase / Firestore（既存の `kakigawatei-franchise` プロジェクトをそのまま使える）**
- 新コレクション `crowd/{shopId}` に1ドキュメント。これだけ。
- **書き込み量**：10店 × 5分間隔 = 2,880 writes/日 → **Firestore無料枠（2万/日）に余裕で収まる**

### アプリ側（`index.html`）
- 既存の**お店ピン／お店ページ**に、丸いランプを1個足すだけ
- `crowd/{shopId}` を読んで色を出す。**改修は小さい**（既存のFirebase REST呼び出しを流用できる）

---

## 3. 🚨 いちばん大事な設計：**古いデータをそのまま出さない**

**PCが落ちる。ネットが切れる。カメラが向きを変えられる。必ず起きる。**

→ **`updatedAt` が10分以上古かったら、色を出さずに ⚪️グレー（情報なし）にする。**

これをやらないと、**「🟢空いてます」と表示されたのに満席**という最悪の事故が起きる。
**古い情報を出すくらいなら、何も出さない方が100倍マシ。**

---

## 4. 法務・プライバシー（ここが本丸）

**「店内をAIが24時間監視」は、扱いを間違えると一発でアウト。** ただし**設計で回避できる**。

### 守るべきこと
1. **映像を店外に送信しない・保存しない**
   → 個人情報（顔＝個人識別符号）を**そもそも取得・保有しない**設計にする
2. **顔認識・個人識別は一切しない**（「人が何人いるか」だけ）
3. **店頭に掲示する**：「混雑状況の計測のためカメラ映像をAIで解析しています（映像の保存・送信は行いません）」
4. **従業員の監視には使わない**（労務トラブルの元。加盟店との契約書にも明記すべき）
5. **お店がいつでもOFFにできる**こと

> **この「映像を外に出さない」設計にすると、"AIカメラ監視"の怖さがほぼ消える。**
> お店にも「**映像はこの箱から一歩も出ません。出ていくのは"3段階の色"だけです**」と説明できる。
> **これは営業トークとしても強い。**

---

## 5. 誤判定リスクと、その回避策

| リスク | 現実に起きること | 対策 |
|---|---|---|
| カメラに映らない席がある | カウンター・個室・2階が死角 | **「全体の混雑」ではなく「見えている範囲の目安」と割り切る** |
| 店員も"人"として数えてしまう | 開店前なのに🟡 | 閾値調整／営業時間外は送信しない |
| 誤って🔴が出る | **客が来ない＝営業妨害になりかねない** | **店側が手動で上書きできるボタンを必ず用意** |
| カメラの角度が変わる | 精度が突然落ちる | 定期的に見直す。おかしければグレーに落とす |

> **⚠️ 一番怖いのは「精度」ではなく「お店の売上に悪影響を出すこと」。**
> だから **お店が最終決定権を持つ**設計（手動上書き・OFF）にする。

## 6. 段階導入（ここが提案の肝）

### 🥇 Phase 0：**カメラなしで先に作る**（数日で実装可能・最重要）
**「店員が3段階のボタンを押すだけ」の混雑ステータス機能**をアプリに載せる。

- お店の管理画面（or LINE/簡易ページ）で「🟢空いてる／🟡やや混雑／🔴混雑」を押すだけ
- アプリのお店一覧・地図ピンにランプが出る
- **AIも、カメラも、PCも要らない。それでも"混雑が見えるアプリ"の価値は9割検証できる。**

**なぜ先にこれをやるべきか：**
- ユーザーが**本当にその情報を求めているか**が分かる（タップ率・来店率）
- **お店が実際に更新してくれるか**が分かる（← ここが最大の不確実性。忙しい店は押さない）
- **もし誰も見ない／誰も押さないなら、AIを作る意味がない**
- **カメラAIは、この「押すのが面倒」を解決する自動化として後から入れればいい**

### 🥈 Phase 1：1店舗でAI実証
- **東京油そば 柿川亭**（既に加盟店・実店舗あり）で1台試す
- Raspberry Pi + 既存カメラ + 人数カウント → Firestore
- **手動ボタンの結果とAI判定を突き合わせて、精度を測る**

### 🥉 Phase 2：横展開＆商品化
- **プレミアムプランの目玉機能**にする（「混雑ランプ対応店」）
- 導入費（機材＋設置）を別途もらう形も可能

## 7. ビジネス的な価値（ここが面白い）

- **ユーザー価値**：「行ったら満席だった」を無くす。**特に長岡花火・イベント時に真価**
- **お店の価値**：**空いている時間に客を誘導できる**（＝機会損失を減らす）
  - 「🟢の店に今行くとクーポン2倍」みたいな**空席誘導キャンペーン**も作れる ← これは相当強い
- **ながおか探検録の価値**：**"歩く"アプリに"今どこが空いてるか"が乗る**＝実用アプリになる
  - ゲームで終わらず、**日常的に開かれるアプリ**になる。これはでかい。

---

## 8. 正直な難所

1. **お店に常時起動のPC/端末を置いてもらう必要がある**（＝ハードルは高い。電源・ネット・置き場所）
2. **既存カメラがRTSPに対応していない**ことがある（古い機種・クラウド専用機）→ 事前調査が必要
3. **メンテナンスは運営持ち**になる（10店舗＝10台の面倒を見る覚悟が要る）
4. 精度は**「目安」以上にはならない**。過信させない見せ方（「混雑の目安」と明記）が必須

## 9. ✅ Phase 0 実装済み（2026-07-14）

### できたもの
- **`crowd.html`** … 加盟店スタッフ用の切替ページ（スマホ想定）
  - URL：`https://kakigawatei.github.io/nagaoka-tanken/crowd.html?shop=kakigawatei`
  - **PIN**は `data/shops.json` の `coupon.staffPIN`（クーポン確認PINと同じ）。一度入れれば端末に記憶
  - 🟢空いています ／ 🟡やや混雑 ／ 🔴混雑しています ／ ⚪️表示を消す の**大きな4ボタン**
  - **ホーム画面に追加**しておけばワンタップで開ける
- **`index.html`** … 混雑ランプの表示
  - **お店ピン**の右上に色付きの丸（🔴は点滅）
  - **お店モーダル**に「🟢 空いています ／ お店からの情報」の行
  - **90秒ごとに自動更新**＋画面復帰時にも取り直す
- **10分ルール**：`updatedAt` が10分以上古い／`level=off` のときは**何も表示しない**（`crowdOf()` が null を返す）

### データ構造（Firestore）
```
crowd/{shopId}
  level:     "green" | "yellow" | "red" | "off"
  updatedAt: timestamp
  source:    "manual" | "ai"     ← ラズパイのAI判定に差し替えても同じdocを書くだけ
```

### ⚠️ 残作業：Firestoreのセキュリティルールを追加する（masaさんの作業）
**これを入れるまで動きません**（現状は書き込みも読み取りも 403）。
Firebaseコンソール → Firestore Database → ルール に、**既存のルールに追記**：

```
match /crowd/{shopId} {
  // 混雑レベルは個人情報ではないので公開読み取りでよい（アプリが認証なしで読む）
  allow read: if true;

  // 書き込みは「認証済み」かつ「決められた3フィールドと決められた値」のみ許可
  allow write: if request.auth != null
    && request.resource.data.keys().hasOnly(['level', 'updatedAt', 'source'])
    && request.resource.data.level in ['green', 'yellow', 'red', 'off'];
}
```

### 🔒 セキュリティ：**店舗ごとのログインに変更した（2026-07-14）**

当初のPIN方式は「**匿名認証さえ通れば、誰でも他店のランプを書き換えられる**」欠陥があった（PINはページ側のチェックなので迂回可能）。
→ **Firebase Authentication のメール／パスワードによる「店舗ごとのログイン」に変更**し、**Firestoreのルールで自分の店しか書けないことを保証**する。

**しくみ（2026-07-14 改訂：メールアドレスは自由にした）**
アカウントと店舗の紐付けを**メールの文字列ではなく `uid`** で行う。
→ **どんなメールアドレスでも使える**（Gmail、お店の実在アドレス、何でもよい）。

```
Firestore: shopOwners/{uid} = { shopId: "kakigawatei" }
                ↑ Authenticationのユーザー UID
```
- **crowd.html**：メールアドレス＋パスワードでログイン → `shopOwners/{自分のuid}` を読んで**自分の店を特定**
- **Firestoreルール**：`shopOwners/{uid}.shopId` と書き込み先の `{shopId}` が**一致するときだけ**書ける
→ **他店のランプは物理的に書けない**

**crowd.html は1つのURLに集約**（店舗ごとのURLは不要）：
`https://kakigawatei.github.io/nagaoka-tanken/crowd.html`
→ ログインすると自動でその店のページになる。次回からは**自動ログイン**（リフレッシュトークンを保存）。

**実在のメールを使う利点**：お店が**自分でパスワード再設定**できる（Firebaseからリセットメールが届く）。架空ドメインだとメールが届かないので、運営が毎回コンソールで再設定することになる。

### 🛠 新しい店舗を追加する手順（運営）
1. **Firebaseコンソール → Authentication → ログイン方法 → 「メール/パスワード」を有効化**（初回だけ）
2. **Authentication → ユーザー → 「ユーザーを追加」**
   - メール：**そのお店の実在アドレス**（例：`info@kakigawatei.com`）／テストなら自分のGmailでもOK
   - パスワード：その店に伝えるもの（**店ごとに違うものにする**）
   - 作成後、**そのユーザーの UID をコピー**
3. **Firestore → データ → コレクション `shopOwners` を作成**
   - **ドキュメントID＝コピーしたUID**
   - フィールド：`shopId`（文字列）＝ `data/shops.json` の店舗ID（例：`kakigawatei`）
4. お店には「**メールアドレス**」「**パスワード**」「**URL**（全店共通）」を伝える

### Firestoreルール（crowd の部分を差し替える）
```
match /shopOwners/{uid} {
  // 自分の紐付けだけ読める（他人のは読めない）。書き込みはコンソールからのみ。
  allow read: if request.auth != null && request.auth.uid == uid;
  allow write: if false;
}

match /crowd/{shopId} {
  allow read: if true;

  // 自分に紐付いた店のドキュメントだけ書ける
  allow write: if request.auth != null
    && get(/databases/$(database)/documents/shopOwners/$(request.auth.uid)).data.shopId == shopId
    && request.resource.data.keys().hasOnly(['level', 'updatedAt', 'source'])
    && request.resource.data.level in ['green', 'yellow', 'red', 'off'];
}
```

> ※ `data/shops.json` の `coupon.staffPIN`（クーポン確認用）は**別物**。こちらは今も全店共通の "1234" なので、**実店舗を増やす前に店舗ごとの値に変える**こと。

### 次の一手：ラズパイ（Phase 1）
`crowd/{shopId}` に `source:"ai"` で書くだけなので、**アプリ側の改修は不要**。
ラズパイ側で：カメラ取得（RTSP/USB）→ 人物検出（YOLOv8n等）→ 人数÷満席人数 → 1〜5分おきに同じdocへPATCH。
**手動ボタンの結果とAI判定を突き合わせて精度を測れる**（同じdocを見ればいい）。

## 10. 結論

- **技術的には十分作れる。既存のFirebase構成にそのまま乗る。**
- ただし **Phase 0（手動ボタン）を先にやるべき**。**AIは"面倒くささを消す自動化"であって、価値の source ではない。**
