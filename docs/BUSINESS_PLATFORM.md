# ながおか探検録 — 地域ビジネス連携プラットフォーム構想（将来設計）

> ⚠️ **【2026-07-24 マネタイズ方針＝通常3プラン＋期間限定キャンペーン割引】** 通常価格：ライト¥3,000／スタンダード¥6,000／プレミアム¥12,000（月額・税込）＋加盟金¥5,000（初回）。**2026年12月31日まで「開店応援キャンペーン」で全プラン割引**＝ライト無料／スタンダード¥3,000／プレミアム¥6,000／加盟金無料。**最低利用期間なし（旧3か月縛りは廃止）。2027年1月以降は加盟状況を見て延長or通常価格を判断（面取り優先）。** 本文中の旧記述（3か月縛り・先行加盟の段階割引・一括前払い等）は旧版で凍結。現行の正式内容は `plans.html`／`terms.html`／`tokushoho.html`（改定済）と営業資料 `docs/merchant/merchant_guide(_compact).html/pdf`。混雑ランプは全プラン無料。

街歩きゲームに地域のお店（飲食店・小売店など）を組み込む、月額サブスク型のローカルマーケティング機能の設計メモ。**今すぐ作るものではなく、後で実装するための設計図**。（2026-07-07 masaさん発案）

---

## 1. コンセプト（三方よし）
- **お店**：月額サブスクで「加盟店」として地図に表示され、来店した探検者に特典（クーポン/引換券/限定サービス）を提供 → 集客。
- **探検者（プレイヤー）**：無料でお得。お店を巡る動機が増え、回遊が楽しくなる。
- **運営（masaさん）**：加盟店サブスクが収益。地域の送客プラットフォームに。

ゲームの「歩く楽しさ」を実店舗送客に変える＝ご当地マーケティング。

## 2. 登場人物（ロール）
- 探検者：既存プレイヤー。お店ピンを巡って特典GET。
- 加盟店：月額課金で掲載。特典を設定、来店/利用状況を把握。
- 運営：加盟店の審査・契約・掲載管理・課金回収。

## 3. 体験フロー
**プレイヤー側**：地図に「お店」ピン（お宝ピンと区別）→ 近づく/来店で引換券を獲得 → 店頭で提示 → スタッフが使用処理。お店限定サービスやスタンプラリーも。
**加盟店側**：トップに「法人・個人店のお客様はこちら」ボタン → 申請フォーム → 運営とやり取り → 契約 → 掲載開始 → 管理画面で特典編集・実績確認。

## 4. 最重要：サーバーレスの限界と「段階導入」
現状アプリは静的（GitHub Pages＋localStorage、サーバー無し）。
- **申請フォームだけ**ならサーバー無しでも可能。
- しかし**アカウント・月額課金・クーポン利用の確実な管理**にはバックエンドが必要。
→ いきなり作り込まず2段階で。

### フェーズA（すぐ着手可・サーバーほぼ不要の“手動運用MVP”）
- 「加盟店はこちら」ボタン → **外部フォーム（Googleフォーム/Tally等）**で申請受付（メール通知）。
- 審査・契約は**オフライン**（請求書 or Stripe決済リンク）。
- 掲載データは **`shops.json`**（pois.json と同じ静的データ）を運営が更新 → デプロイで反映。
- クーポンは**静的**（お店ごとに固定特典）。利用は「画面提示＋スタッフ用PINタップで“使用済み”表示」。利用記録は端末内（低額特典向けの簡易方式）。
- ねらい：**今の構成のまま最短で開始**。まず1〜3店舗で実証。

### フェーズB（拡張・バックエンド導入で本格プラットフォーム）
- Firebase / Supabase 等で：お店/プレイヤーのアカウント、掲載の自己管理、**サブスク自動課金（Stripe Billing）**、クーポン発行・利用の**サーバー検証（重複・不正防止）**、来店/利用の分析ダッシュボード。
- **本物の地図（Leaflet / MapLibre）**でお店をリアル位置表示。営業時間・在庫連動なども。
- 解約・返金・監査など運用が堅牢に。

## 5. データモデル（追加イメージ）
```
shop = {
  id, name, category, districtId, lat, lng,
  plan,            // ライト/スタンダード/プレミアム
  status,          // active / paused
  offer:{ title, desc },
  coupon:{ title, terms, expiry, perUserLimit, redeemPIN },
  links:{ web, sns, tel }, hours
}
// プレイヤーsave（既存tekuteku_saveに追加）
redeemedCoupons: [ { shopId, at } ],
shopVisits: { shopId: { count, lastDate } }
```
フェーズAは `shops.json`、フェーズBはDB化。

## 6. マップ統合
- 地図に「お宝」レイヤーと「お店」レイヤー（トグル/フィルタ）。お店は専用ピン。
- 加盟店詳細：特典・クーポン・アクセス・営業時間・SNS。
- 「今日の得するお店」等のレコメンド枠。

## 7. クーポン/引換券の設計
- 種類：来店特典、初回限定、期間限定、スタンプN個で1枚 など。
- 発行：来店（GPS）で自動 or その場でボタン。
- 利用：店頭でプレイヤーが「使う」→確認→**スタッフPIN or 日替わりコード**で確定→使用済み表示（スクショ悪用対策に時刻表示/アニメ/コード）。
- フェーズA＝端末内記録（簡易）、フェーズB＝サーバー検証（確実）。

## 8. 課金・契約
- **プラン（2026-07-14 確定・すべて税込）**：ライト¥3,000（ピン＋お店ページ＋クーポン1種）／スタンダード¥6,000（＋オリジナル図鑑カード：3回来店で豪華版に昇格・★付与）／プレミアム¥12,000（＋店舗独自イベント〈ヒアリング→アップデートで実装〉＋複数クーポン＋特別仕様の豪華カード＋**スポンサー表記**＋**AI導入サポート（おまけ）**＋上位表示・レポート）。初期登録料¥5,000（税込・初回のみ）。すべて月額。詳細は `MERCHANT_APPLICATION_FORM.md`、営業資料は `docs/merchant/`。
- 決済：フェーズA＝Stripe決済リンク/請求書、フェーズB＝Stripe Billingで自動継続・解約。
- **契約書・利用規約・特商法表記・プライバシーポリシー**が必要（法人相手は特に）。

## 9. 法務・信頼・運用
- 個人情報（申請フォーム）・決済情報の取り扱い、規約/プライバシーポリシー整備。
- クーポン表記の適正（景表法）、加盟店の審査基準。
- 運営体制（問い合わせ・掲載修正・解約対応）。

## 10. 収益とKPI
- 収益：加盟店月額 × 店舗数（＋上位プラン）。
- KPI：加盟店数、クーポン発行/利用数、送客数、継続率。
- 好循環：プレイヤーが得する → 回遊増 → お店の効果増 → 加盟店増。

## 11. おすすめの第一歩（最小実証）
1. トップに「法人・個人店のお客様はこちら」ボタン ＋ **Googleフォーム申請**（サーバー不要）を設置。
2. `shops.json` ＋ お店ピン ＋ 固定クーポン（PIN確定）を **1〜3店舗**で試験。
3. 反応を見て フェーズB（バックエンド/課金自動化）へ判断。

## 13. 収益の2軸（2026-07-07 masaさん補足）
運営収益は2本立て：
- **(a) スポンサー月額（B2B・基本）**：加盟店サブスク。前述のとおり。
- **(b) ユーザー課金（IAP・補助）**：プレイヤーがゲーム内コインやアイテムを購入して、ゲームを“加速”できる。
  - **大原則：無課金でも十分に楽しめる設計（pay-to-winにしない）**。課金は「時短・便利・見た目（装飾）」に限定し、ゲームバランスを壊す“強さ”は売らない。
  - 商品例：コインパック、便利アイテム（探知強化/ヒント/ダブルコイン券の有償版）、装飾・称号枠など。

### 重要（技術）：有償コインには「サーバー側アカウント」が必須
現状コインは端末内（localStorage）＝簡単に書き換え可能。**実際のお金でコインを売る瞬間から、コイン残高はサーバーで管理（ユーザーアカウント＋DB）する必要**がある（でないと不正・水増しし放題）。
→ ユーザー課金は事実上**フェーズB（バックエンド＋決済）と同時**に入る。スポンサー課金の基盤（Stripe等）と共用できるので相性は良い。
- 決済：Web/PWAなら Stripe 等で直接（アプリストアの30%手数料を回避）。将来ネイティブアプリ化するとストア課金（手数料）になる点に注意。

## 14. クーポン不正防止は「小さく始めて拡大」（masaさん方針）
- 最初：受けられるのは**小さなサービス**（一杯サービス、少額割引等）。端末内記録の簡易方式でOK。
- 慣れてきたら：**大きめのコイン消費**や価値の高い特典も。ここは**サーバー検証（フェーズB）**で不正を防ぐ。
- 段階を踏むほど、必要な堅牢さ（アカウント・使用履歴・重複防止）が上がる＝バックエンド前提。

## 15. 最終目標：コインで“商品”が買える（後で本格検討・要法規制チェック）
コインで実際の商品/サービスが買えるところまで見据える。実現可能だが、日本では法規制に注意：
- **資金決済法（前払式支払手段）**：ユーザーが“お金で買ったコイン”が商品と交換できると、前払式支払手段に該当しうる。未使用残高が基準額（期末2,000万円）超で財務局への届出・供託等の義務が生じる。
- **景品表示法**：無料で貯めたコインを景品的に商品交換する場合、景品類の価値上限などの規制。
- 実務的な整理：**「買ったコイン」と「歩いて貯めたコイン」を分ける**、交換の価値に上限を設ける、商品はお店（加盟店）が提供する形にする——等でリスクを抑える設計が定石。
- → この段階に進む前に、規制の確認（必要なら専門家相談）をおすすめ。**急がず、まず“小さな特典”から**。

## 16. 未確定（要相談）
- ~~サブスクはB2Bか~~ → **確定：基本はスポンサー月額。加えてユーザー課金（IAP）も入れる**。
- クーポン確実性 → **確定方針：小さく始めて段階的に堅牢化**。
- ユーザー課金を入れる＝アカウント＋バックエンド＋決済が必要。**いつフェーズBへ踏み込むか**（＝どのタイミングでサーバーを用意するか）。
- **サーバーの用意先**：masaさんの既存ホームページのサーバー/ドメインが使えるか要確認（PHP+DB可のレンタルサーバーならバックエンドを載せられる。静的/ノーコードならFirebase/Supabase＋サブドメイン）。
- 実地図（Leaflet）導入のタイミング。運営主体・決済事業者・規約/資金決済法対応の体制。
