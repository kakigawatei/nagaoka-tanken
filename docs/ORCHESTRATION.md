# モデル・オーケストレーション方針（Fableトークン節約）

このプロジェクトの最重要方針。**高価な Fable のトークン消費を最小化しつつ、継続的に開発を進める**ための運用ルール。

---

## なぜ節約できるのか（原理）
Claude Code でサブエージェント（別モデル）に作業を委譲すると：
- そのサブエージェントが読むファイル・書くコードなど**大きな入出力は、サブ側のモデル（Sonnet/Haiku）で課金**される。
- Fable のコンテキストには**短い要約だけ**が戻る。

→ つまり「Fable が自分で大きく読む・大きく書く」のをやめ、**Fable は指示と要約レビューだけ**にすれば、Fable のトークンは劇的に減る。

## 2つのオーケストレーション形態
**(A) 人間側で切替（最大の節約・推奨のデフォルト）**
普段の開発セッションは `/model` で **Sonnet** にしておく。設計判断・難所のときだけ **Fable / Opus** に切り替える。ルーチン作業に Fable を一切ロードしない＝Fableトークンほぼゼロ。

**(B) セッション内委譲（Fableが司令塔のとき）**
Fable セッションのまま、重い作業を Sonnet/Haiku のサブエージェントに委譲（背景実行）。Fable は指示書と要約だけ扱う。

> 実運用は (A) を基本に、設計レビュー回だけ (B) を併用するのが最もコスパが良い。

## 役割分担
| モデル | 役割 | 使いどころ |
|---|---|---|
| **Fable**（司令塔・最小限） | 要件解釈、設計判断、委譲の指示書作成、要約レビュー、方向修正 | アーキテクチャ／方針決定のみ。大きなファイルを自分で読まない・貼らない |
| **Opus** | 難所の設計、複雑なデバッグ、リファクタ設計 | 本当に難しい時だけ |
| **Sonnet** | 実装全般（機能追加・UI・修正・テスト） | **既定の実働**。ほとんどの開発はここ |
| **Haiku** | 機械的作業（POIデータ入力、緯度経度・住所・史実の調査、素材リネーム、定型テスト、grep調査） | 量産・雑用 |

## 委譲ループ
1. Fable が依頼を受ける
2. タスクを**小さく自己完結な単位**に分割
3. 各単位を適切なモデルのサブエージェントに**指示書付きで委譲**（背景実行）
4. サブが実装・検証し、**短い要約**を返す
5. Fable は要約だけ見て次を判断（往復＝Fable消費を最小化）

## Fableのトークンを抑える具体ルール
1. **ソースの全文読み込み・全文貼り付けを Fable でやらない**（読むなら Haiku に要約させる）。
2. 指示書に**受け入れ条件（Doneの定義）を明示**し、往復を減らす。
3. **まとめて大きめの単位**で委譲し、オーケストレーション回数を減らす。
4. 設計書 / ROADMAP / プロジェクトメモリを**唯一の共有記憶**に。再オリエントは小さな読みで。
5. **Opus は本当に難しい時だけ**。既定は Sonnet、雑用は Haiku。

## 委譲タスク指示書テンプレ（コピペ用）
```
【対象】nagaoka-walk（C:\Users\masat\Desktop\nagaoka-walk）
【ゴール】<何を完成させるか。1〜2文>
【変更範囲】<触ってよいファイル／触らないファイル>
【受け入れ条件(Done)】
  - <箇条書きで検証可能な条件>
  - 既存の設計書（docs/DESIGN.md）に沿うこと
【参照】docs/DESIGN.md, docs/ROADMAP.md, data/pois.sample.json
【報告】完了時は「変更点・確認方法・残課題」を5行以内で要約して返す
```

## Claude Code での頼み方（例）
- (A) 端末で `/model` → Sonnet に。普段はこれで開発。
- (B) Fableセッション中：「このタスクを **Sonnet のサブエージェント**で実装して。指示書テンプレに沿って」と頼めば、Fable が上記ループで委譲する。
- 調査系：「POIの緯度経度を **Haiku のサブエージェント**で調べて `data/pois.json` に埋めて」。
