# お宝カードのシリアルナンバー／限定企画（サーバー連携）設計メモ

masaさん発案（2026-07-10）。お宝カードに通し番号を付け、「あなたは◯番目の獲得者です」を
表示。早い者勝ちの面白さ＋「限定◯枚」チャレンジ企画を可能にする。

## これは何が必要か
現在のアプリは **完全サーバーレス**（localStorage・GitHub Pages）。
シリアルナンバーは「全ユーザー共通のカウンター」を正確に増やす必要があるため、
**バックエンド（サーバー）が必須**。＝この機能でアプリに初めてサーバーが入る。
※将来の「お店連携フェーズB（クーポン検証）」も同じバックエンドで実現できる＝相乗効果あり。

## 推奨構成：Firebase（Firestore + 匿名認証）
- **匿名認証（Anonymous Auth）**：端末ごとに安定したIDを自動発行。**ログイン不要**を維持できる
  （＝「誰でもすぐ遊べる」の原則を崩さない）。
- **Firestore のトランザクション／atomic increment**：
  カード初回発見時に `cards/{cardId}.count` を +1 して、その番号を採番＝重複しない通し番号。
- 採番した番号はユーザーのデータ＋localStorageに保存し、発見演出と図鑑カードに表示。

### データ構造（案）
```
cards/{cardId}         : { count: 123 }                // 全体の獲得数（カウンター）
users/{uid}/cards/{id} : { serial: 7, at: <timestamp> } // その人の採番結果
campaigns/{id}         : { cardId, limit, start, end }  // 限定企画（任意）
```

### 採番フロー
1. 初回発見 → オンラインなら Firestore トランザクションで count+1 → serial 取得
2. 「あなたは **◯番目** の発見者！」を発見モーダルに表示、図鑑カードにも「No.◯」バッジ
3. オフライン時は localStorage に「採番待ち」で記録 → 次回オンラインで確定

## 限定企画（フェーズ2）
- `campaigns` に `limit`（例：先着100枚）・`start`/`end`（期間）を設定
- count >= limit で「**限定100枚 受付終了**」表示、そのカードは以降入手不可（or 別デザイン）
- 「早い者勝ち」ランキング（上位◯人の称号）なども拡張可能

## コスト
- Firebase 無料枠（Spark）で小規模は十分（Firestore 読み書き 各5万/日が無料枠の目安）。
- 規模が伸びても従量課金で安価。まず無料枠で開始可能。

## 正直な注意点（要相談）
1. **端末ごとのID**：匿名認証は端末単位。機種変更・再インストールでIDが変わり、
   再取得できてしまう。カジュアル用途なら許容、厳密化はアカウント連携（後日）で対応。
2. **なりすまし対策**：GPS発見はクライアント判定なので、位置偽装で採番される余地がある。
   通常カードは実害小だが、「限定」企画では価値が出るぶん不正の動機も生まれる。
   本気の限定企画をやるなら **サーバー側でGPS検証・レート制限**（フェーズ3）が要る。
3. **オフライン整合**：オフライン発見時の採番タイミング設計（上記フロー）。

## 段階導入（おすすめ）
- **フェーズ1（MVP）**：Firebase匿名認証＋Firestoreで「通し番号採番＋表示」。まず体験を作る。
- **フェーズ2**：限定企画（limit/期間）＋受付終了表示＋早い者勝ち要素。
- **フェーズ3**：不正対策（サーバーGPS検証）＋任意アカウント連携（機種変で引き継ぎ）。
